# Implementation Plan: Live Data for Road Assets Demo

This document outlines the step-by-step technical plan to replace the hardcoded demo data in the Road Assets Demo Page (`/demo/road-assets`) with live, functional API calls to the platform's backend and worker outputs.

## Current State vs. Target State

| Component | Current (Hardcoded) State | Target (Live) State |
| :--- | :--- | :--- |
| **Point Cloud Data** | 8,000+ lines of static arrays in `pointCloudData.ts` and `pointCloud3DData.ts`. | Streamed dynamically from the `copc_url` in Cloudflare R2 using the existing `CesiumViewer` component. |
| **Vector Assets** | `buildDemoFeatures()` generating static GeoJSON objects in memory. | Fetched from the `road_assets_url` (GeoJSON file generated by the `road-assets` worker). |
| **Statistics Panel** | Hardcoded strings (e.g., "28,281 Total Points", "4.1s total"). | Driven by `dataset.point_count` and `dataset.road_asset_stats` JSONB column. |
| **Pipeline UI** | Static list of steps that users click through manually. | Driven by the realtime `processing_jobs` status and progress percentage. |

---

## Phase 1: API & Data Fetching Layer

The backend already produces the required data (via the Tiling and Road Assets workers), but the frontend needs to fetch it dynamically.

### 1. Update the Page Route to Accept a Dataset ID
The demo page should be able to view real datasets. We will modify `frontend/app/demo/road-assets/page.tsx` to accept a `datasetId` query parameter, falling back to a known "demo" dataset ID if none is provided.

```typescript
// frontend/app/demo/road-assets/page.tsx
export default function RoadAssetsDemoPage({ searchParams }: { searchParams: { id?: string } }) {
  const datasetId = searchParams.id || process.env.NEXT_PUBLIC_DEFAULT_DEMO_DATASET_ID;
  return <RoadAssetsDemoClient datasetId={datasetId} />
}
```

### 2. Implement Data Fetching in the Client
Inside `RoadAssetsDemoClient.tsx`, replace the static imports with API calls:

1.  **Fetch Dataset Metadata:** Call `GET /api/v1/datasets/{datasetId}` to retrieve the `copc_url`, `road_assets_url`, and `road_asset_stats`.
2.  **Fetch GeoJSON:** If `road_assets_url` exists, fetch the GeoJSON file directly from the public R2 bucket.
3.  **Fetch Job Status (Optional but recommended):** If the dataset is still processing, poll or subscribe to the `processing_jobs` table to animate the pipeline UI.

---

## Phase 2: Replacing the 3D Viewer

The current `PointCloud3DViewer.tsx` uses raw Three.js to render static arrays. The platform already has a robust, production-ready `CesiumViewer.tsx` capable of streaming massive COPC files.

### 1. Swap the 3D Component
Remove `PointCloud3DViewer.tsx` entirely. In `RoadAssetsDemoClient.tsx`, when `viewMode === '3d'`, render the `CesiumViewer`:

```tsx
{viewMode === '3d' && dataset?.copc_url && (
  <CesiumViewer 
    copcUrl={dataset.copc_url} 
    bounds={dataset.bounding_box}
    roadAssetsGeoJSON={geoJsonData} // Requires updating CesiumViewer to accept GeoJSON
  />
)}
```

### 2. Add GeoJSON Support to CesiumViewer
Update `CesiumViewer.tsx` to accept a `roadAssetsGeoJSON` prop. Use Cesium's `GeoJsonDataSource` to drape the road markings, kerbs, and traffic signs over the 3D terrain/point cloud.

## Phase 3: Upgrading the 2D Canvas

If the lightweight 2D canvas is retained for the split-view experience, it must render the live GeoJSON rather than the hardcoded `buildDemoFeatures()`.

### 1. Render Live GeoJSON
Map the fetched GeoJSON `FeatureCollection` to the canvas drawing logic. The current drawing logic already expects a `GeoFeature` array, so mapping the fetched data is straightforward:

```typescript
// After fetching GeoJSON from road_assets_url
const features = fetchedGeoJson.features.map(f => ({
  type: 'Feature',
  geometry: f.geometry,
  properties: f.properties
}));
setLiveFeatures(features);
```

### 2. Handle the Point Cloud Backdrop
The 2D canvas currently draws raw points from `POINT_CLOUD_DATA`. Since downloading a full `.laz` file to the browser just to draw a 2D backdrop is inefficient, the recommended approach is:
*   **Option A:** Use a 2D mapping library like Leaflet (via the existing `MapPanel.tsx`) instead of a custom HTML canvas.
*   **Option B:** Have the tiling worker generate a lightweight 2D preview image (e.g., an orthophoto) and draw that onto the canvas background.

*Recommendation:* Replace the custom 2D canvas entirely with the platform's existing `MapPanel.tsx`, passing the GeoJSON to it. This unifies the codebase.

---

## Phase 4: Dynamic UI & Statistics

The sidebar UI must be wired to the real dataset properties.

### 1. Wire the Stats Panel
Update the "Dataset Summary" grid to use the live data:
*   **Total Points:** `dataset.point_count?.toLocaleString() || 'N/A'`
*   **Assets Found:** `dataset.road_asset_stats?.total_features || 0`
*   **Road Area / Length:** Extract from `dataset.road_asset_stats` if available.

### 2. Wire the Layer Toggles
Dynamically generate the layer toggles based on the unique `asset_type` properties found in the live GeoJSON, rather than a hardcoded list. Update the counts next to each layer name by filtering the live features array.

### 3. Wire the Pipeline UI
Instead of a static list of steps, fetch the `processing_jobs` record for this dataset where `job_type = 'road_assets'`. 
*   If `status === 'completed'`, show all steps as green/checked.
*   If `status === 'processing'`, use the `progress` integer (0-100) to animate the active step.
